1. Problem Statement & Business Context

BloomDirect is a fictional floral e-commerce business (inspired by a real business) shipping approximately 1,000 orders per week via three carrier methods: UPS Ground, FedEx Ground, and FedEx Air. With a ~10% shipping failure rate, roughly 100 shipments per week arrive late, damaged, or go missing — each representing a recoverable loss through carrier guarantee claims.

The Core Problem: Filing carrier claims is a multi-step process involving tracking verification, eligibility checking, claim drafting, email submission, follow-up on rejections, and portal-based FedEx filing. Doing this manually for 100 failures/week is infeasible for a small operations team — most claims simply go unfiled.

Key Design Decision: 98% Deterministic, 2% LLM

Classification, SLA checking, probability scoring, and routing are all handled by deterministic Python code. The LLM (Claude claude-sonnet-4-6) is only invoked for two tasks: (1) drafting claim emails, and (2) analyzing rejection responses. This keeps LLM cost at ~$5–15/month for ~60 claims/week, while maintaining full auditability of all routing decisions.

3. Sub-Agent Descriptions

Agent 1 — MCP 1: Carrier Tracking MCP Server

MCP Server Deterministic Tool: UPS Shippo API + FedEx Track API

Role: Fetches real-time tracking events for every non-delivered shipment. Returns structured tracking history (timestamp, location, event code, description).

Cache Strategy: tracking_cache table stores all responses permanently. If delivered=True is already cached, the MCP call is skipped entirely — dramatically reducing redundant API calls. The first bad event in the history trail determines claim type.

Tools: Shippo /tracks/ups/{id}, FedEx Track API (OAuth), mock fallback in test mode.

Prompt Strategy: No LLM involved — pure deterministic API call + SQLite caching logic.

Agent 2 — Claim Reasoner (Classification & Probability)

Deterministic No LLM tokens consumed

Role: Reads the tracking history from MCP 1 and deterministically classifies each failure. Uses a first-bad-event-wins rule to assign claim type.

Event PatternClaim TypeBase Probability
Mechanical failure / Missed flightCARRIER_DELAY0.75 (golden cases)
Damage detected in trailDAMAGE0.65
No delivery event, no scansLOST0.60
Weather eventCARRIER_DELAY0.35
Address error (our fault)NO CLAIM—
Refused + no damageNO CLAIM—
Delivered + lateLATE0.55

HITL Triggers: Unknown failure pattern, probability <0.3 after rejection, high-value claims, borderline eligibility.

Agent 3 — Claim Drafter (LLM)

LLM: Claude claude-sonnet-4-6 ~60 invocations/week

Role: Drafts professional, carrier-appropriate claim emails based on tracking history, failure type, claim probability, and tone setting.

Prompt Strategy: Few-shot prompting with 3 examples (firm tone, balanced tone, first attempt). CoT reasoning step before draft. Structured output: {subject, body, tone_used, confidence}. Prompt stored in prompts/claim_drafter_prompt.md.

Tone Ladder: Attempt 1 = professional; Probability ≥0.6 on resubmit = firm; Probability 0.3–0.6 = balanced. Language intentionally avoids "formally demand" or "within 5 business days" phrasing.

FedEx Exception: FedEx claims use a deterministic 295-character Excel template — LLM is skipped entirely for FedEx to reduce cost and ensure portal compatibility.

Agent 4 — MCP 2: Email Claims MCP Server

MCP Server Deterministic Tool: Gmail SMTP

Role: Sends drafted claims to the appropriate carrier address. Implements paced sending (10 emails/hour) to avoid spam triggers. Priority-based queue: urgency → LLM draft quality → probability DESC.

Routing: UPS → support@shippo.com; FedEx → portal batch (Layer 1) via Excel download. FedEx is skipped in email paced sender entirely.

Modes: Test (redirect to sandbox email, orange badge, CC disabled) vs Production (real carrier, green badge, CC logistics@arabellabouquets.com).

Agent 5 — Follow-Up & Escalation (LLM)

LLM: Claude claude-sonnet-4-6 Triggered on rejection

Role: Monitors the Gmail inbox via response polling (hourly). When a carrier rejection is received, it extracts rejection reason, recalculates probability, and routes accordingly.

Follow-Up Timeline: Day 10 follow-up if no response; Day 14 follow-up; Day 15 → route to HITL. Filing window enforcement: ≤2 days remaining → auto-file, bypass HITL.

Resubmission Logic: Probability ≥0.6 → auto-resubmit firm tone; 0.3–0.6 → auto-resubmit balanced; <0.3 → HITL queue.

HITL as Ultimate Guardrail: The Human-in-the-Loop queue acts as a final safety net. Any claim where the system has low confidence, unknown failure pattern, or the draft looks suspicious is routed here before sending. Humans see the draft, can edit it, and choose to Send / Skip / Close.

5. Orchestration Pattern

Pattern: Sequential Pipeline with Conditional Routing + HITL Loop

Stage 1 · Data Ingestion
Order API Pull
Fetch invoiced orders for date range → filter not-delivered shipments
Stage 2 · MCP 1 (Tool)
Carrier Tracking
Fetch tracking history → cache-first → identify first bad event
Stage 3 · Conditional Routing
Claim Classification & Probability
Deterministic classification → assign claim type → calculate probability → route: auto-file / HITL / no-claim
Stage 4 · LLM Agent
Claim Drafting (Claude)
Few-shot CoT prompt → generate subject + body → validate output → queue for sending
Stage 5 · MCP 2 (Tool)
Paced Email Sending
Priority queue → 10 emails/hour → UPS via Shippo / FedEx via portal batch
Stage 6 · Loop / Iteration
Follow-Up & Resubmission
Poll inbox → detect rejection → recalculate probability → auto-resubmit or HITL
Stage 7 · HITL Checkpoint (Bonus)
Human Review Queue
Generate Draft → Review/Edit → Send / Skip / Close — 3-step UI flow

Why This Pattern?

The sequential pipeline reflects the real-world claims workflow — each step depends on the previous. Conditional routing is essential because not every shipment needs an LLM (address errors → no claim). The loop pattern mirrors the resubmission reality (99% first-attempt rejection rate). HITL provides human oversight for high-stakes or ambiguous cases, satisfying the bonus requirement.

🌸 BloomDirect — The Claims Recovery Problem

Business Case Study · ISB AMPBA Batch 24 · Group 8

The Business

BloomDirect is a floral e-commerce business (fictional case inspired by a real business) that ships approximately 1,000 orders per week across India and to international destinations. The company uses three primary shipping methods: UPS Ground, FedEx Ground, and FedEx Air, all routed through Magento-based order management. Orders range from everyday bouquets ($30–$60) to premium arrangements for weddings and corporate events ($150–$300+).

Flowers are perishable. A two-day shipping delay doesn't just mean a late delivery — it means a dead product, a disappointed customer, and a lost relationship. The business has built a reputation on timely, beautiful deliveries, and every shipping failure is a direct threat to that promise.

The Problem

With approximately ~10% of shipments failing each week — arriving late, damaged, or going missing — BloomDirect faces around 100 shipping failures per week. Each of these failures represents a recoverable financial loss through carrier guarantee claims.

However, the claims process is painful:

  • Each claim requires checking tracking history, verifying eligibility, drafting a carrier-specific email, and submitting via the right channel
  • UPS claims go through Shippo (the shipping aggregator), not UPS directly
  • FedEx requires portal-based batch filing — email claims bounce
  • 99% of first-attempt claims are rejected — every claim needs follow-up and resubmission
  • The filing window is only 15 days from the ship date — after which claims are permanently ineligible
"We know claims exist. We know we're losing money. But between running the business and managing operations, nobody has time to chase carriers through a 4-step claims process for every delayed shipment. Most claims just go unfiled."
The Scale of the Loss

At ~100 failures/week and an average shipping cost of $15–$30 per shipment, BloomDirect is potentially leaving $1,500–$3,000 per week in recoverable carrier claims on the table — over $75,000–$150,000 per year in unrecovered losses. Even with a 30% recovery rate (realistic for resubmitted claims), that's $20,000–$45,000/year in recoverable revenue.

Why Manual Doesn't Work

The operations team tried manual claim filing. The results were predictable:

  • Claims filed sporadically — no systematic coverage
  • Rejections went unanswered (no bandwidth for follow-up)
  • Filing window expired before anyone noticed
  • FedEx portal batch process was discovered only after months of bounced emails
  • No visibility into which claims were filed, rejected, or recovered
Why Agentic AI is the Right Solution

This problem has a clear structure: defined inputs (order data, tracking events), deterministic rules (SLA policies, eligibility windows), and repetitive execution (draft, send, follow-up, resubmit). It is exactly the type of workflow where a multi-agent system excels.

The key insight is that 98% of the logic is deterministic — classification, probability scoring, routing, and follow-up scheduling can all be handled by code. The LLM (Claude claude-sonnet-4-6) is only needed for two tasks: drafting the claim email in the right tone, and analyzing rejection responses. This keeps LLM costs at $5–15/month for the full system.

Human oversight (HITL) is preserved for the cases that genuinely require judgment: unknown failure patterns, borderline eligibility, high-value claims, and low-confidence resubmissions. The system doesn't replace human judgment — it handles the 80% of cases that don't need it, and routes the 20% that do.
